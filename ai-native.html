The loop never stops: Measure ‚Üí Find issues ‚Üí Speak (describe fix) ‚Üí Generate ‚Üí Review ‚Üí Ship ‚Üí Measure again

01

Speak

Owner: Phillip (Product)

Describe the feature, bug fix, or change in plain English. Be specific about intent, user benefit, and any constraints. This description kicks off the entire cycle.

Responsibilities

  • Write clear feature descriptions with user context
  • Define what success looks like
  • Prioritize what gets built and when
  • Make final go/no-go shipping decisions

Inputs to AI

  • Plain English feature description
  • Reference to similar existing screens
  • User stories or problem statements
  • Any specific constraints or requirements

Output

  • Feature request that triggers AI generation
02

Generate

Owner: AI (Claude Code)

AI reads the entire codebase, understands patterns, and generates backend + iOS code that matches our existing architecture.

What AI Reads

  • CLAUDE.md ‚Äî Master instructions for AI
  • Colors.swift, Typography.swift ‚Äî Design tokens
  • /Components ‚Äî Reusable UI components
  • Existing screens ‚Äî Patterns to follow

Why Documentation Matters

  • AI only knows what's in the repo
  • Code IS the documentation (not separate docs)
  • Better documentation = better generation
  • If it's not pushed, it doesn't exist for AI

Output

  • Pull Request with backend + iOS code
03

UX Review

Owner: Tea & Tim (UX & Design)

Review the actual built UI‚Äînot mockups‚Äîand judge if it matches brand, feels right, and provides intuitive UX.

Responsibilities

  • Review AI-generated UI against brand standards
  • Judge if the UX is intuitive and flows well
  • Catch visual issues (spacing, alignment, color)
  • Approve or request adjustments

How to Give Feedback

  • Direct to AI: "Make the button bigger" works
  • GitHub PR comments: Inline on specific code
  • Slack with screenshot: Circle what's wrong
  • Reference existing screens: "Like the portfolio page"

Output

  • Approve ‚Üí Moves to Eng Review
  • Request Changes ‚Üí AI regenerates
04

Eng Review

Owners: Luka (Backend) + Rok (iOS)

Backend and iOS engineers review code simultaneously. You are the quality gatekeepers‚Äîno code ships without your approval.

Luka

Backend
  • Review API endpoints
  • Check database queries
  • Verify business logic
  • Approve or request changes

Rok

iOS
  • Review SwiftUI code
  • Check view architecture
  • Verify native feel
  • Approve or request changes

Shared Responsibilities

  • Review ALL code pushed to your repo
  • Catch edge cases and potential bugs
  • Ensure it follows our patterns
  • Tell AI what to fix‚Äîit will regenerate
05

Ship

Owner: Everyone

After UX and Eng approval, merge the PR and deploy. What used to take 2-3 weeks now takes 2-3 days.

Checklist Before Merge

  • UX has approved the visual/UX
  • Luka has approved backend code
  • Rok has approved iOS code
  • Tests pass (if applicable)
  • Phillip confirms it solves the user problem

Speed Comparison

  • Before: 2-3 weeks per feature
  • After: 2-3 days per feature
  • Bug fixes: Minutes, not hours
06

Measure

Owner: All Stakeholders

Every Monday, AI generates a report for the team. We review metrics, identify issues, and decide what to iterate on next.

Monday Report Includes

  • Product health: Orders, users, completions
  • Issues detected: Bugs, slow queries, errors
  • AI-proposed fixes: Ready for human review
  • Decisions needed: What requires human input

What We Discuss

  • Approve or reject AI-proposed fixes
  • Prioritize next features
  • Identify patterns in user feedback
  • Decide: iterate, pivot, or delete

Then the Loop Restarts

  • Issues found ‚Üí Phillip speaks new fix ‚Üí Generate ‚Üí Review ‚Üí Ship ‚Üí Measure
